01. Objectives

  • Enable Interoperability: Bridge Java, .NET, and Python.
  • Standardize Communication: Use HTTP/HTTPS protocols.
  • Promote Reusability: Share logic across platforms.
  • Integrate Systems: Connect to external services (Payment/Maps).
  • Ensure Scalability: Implement loose coupling.

02. APIs

An API is the general interface for interaction. A Web Service is a specific API accessed over a network.

03. REST Architecture

REST is an architectural style, not a strict protocol, that uses standard HTTP methods (GET, POST, PUT, DELETE) to interact with resources identified by unique URLs (URIs). It is lightweight and widely used for modern web and mobile applications due to its simplicity and performance HTTP methods:

04. SOAP Protocol

SOAP is a more rigid, protocol-based approach that relies on XML for its messaging format. It is transport-independent but typically uses HTTP or SMTP for communication, often preferred in enterprise scenarios with strict contracts and security requirements (WS-Security).

Note: SOAP examples are more complex due to the extensive XML structure and usually require specific libraries (like zeep in Python) to handle WSDL files and SOAP envelopes.

05. JSON Example

JSON is a lightweight, human-readable, and language-independent text format that is the de facto standard for REST APIs

06. HTML / JS Parsing

Converting JSON strings to objects:

const jsonString = '{"name": "John", "age": 30}'; const user = JSON.parse(jsonString); console.log(user.name); // John

07. XML Format

XML uses a tag-based structure to define, transmit, and validate data between applications, making it universally understood by various programming languages. It is verbose compared to JSON.

08. RESTful Principles

  • Platform Independence: The client doesn't care about the server's OS.
  • Service Evolution: Change the backend without breaking the frontend.

Uses HTTP for transport and JSON for exchange.

09. Resource URIs & Naming Conventions

In a RESTful web API, a resource is an object or service that the client can access. Each resource must have a unique identifier (URI). Well-designed URIs improve API intuitive use and maintainability.

1. Use Nouns, Not Verbs

The URI should identify the resource itself, not the action being performed. The action is determined by the HTTP Method (GET, POST, PUT, DELETE). Avoid "operation" names in your paths.

✗ BAD: /get-customers or /createNewOrder ✓ GOOD: /customers or /orders

2. Pluralization for Collections

Use plural nouns for collection URIs to maintain consistency. This makes it clear whether you are interacting with a list or a specific item.

  • /customers : Refers to the entire collection of customers.
  • /customers/5 : Refers to a specific customer with ID 5.

3. Representing Hierarchy & Relationships

When resources are related, use the URI structure to show that hierarchy. This provides a logical navigation path for developers.

Standard Pattern: /collection/{id}/sub-collection/{id}
  • /customers/5/orders : Retrieves all orders belonging specifically to customer #5.
  • /customers/5/orders/12 : Accesses order #12, verifying it belongs to customer #5.

4. Filtering, Sorting, and Searching

Don't create new URIs for simple variations of a resource. Use Query Strings (parameters) to filter or sort the data returned from a collection.

FILTER: /customers?region=europe&status=active SORT: /orders?sort=date_desc SEARCH: /products?search=laptop

5. Interface Consistency

Ensure that URIs are consistent across the entire API. If a user is identified by a numerical ID in one endpoint, don't use a "username" for the same resource type elsewhere. The naming style (like kebab-case or camelCase) should also be uniform throughout the API.

Important: A REST API should be an abstraction of your data. Avoid mapping URIs directly to your internal database table names to prevent security leaks and maintain flexibility.

10. Performance Tips

Avoid Chatty APIs. Model business entities, not database tables.

Tip: Use pseudo-resources for logic like calculations:
/add?operand1=99&operand2=1

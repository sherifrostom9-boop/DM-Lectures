API Development Lecture 4

RESTful Web API Methods

HTTP Methods, MIME Types, Pagination, HATEOAS, Versioning, and Advanced Patterns

📘 Introduction

RESTful web API methods align with the request methods and media types defined by the HTTP protocol. This lecture covers the most common request methods and the media types used in RESTful web APIs, along with advanced patterns for building production-ready APIs.

💡 Key Topics Covered:
  • HTTP request methods (GET, POST, PUT, PATCH, DELETE)
  • Resource MIME types and content negotiation
  • Asynchronous operations
  • Pagination and filtering
  • HATEOAS implementation
  • API versioning strategies
  • Multitenancy and distributed tracing

🔧 HTTP Request Methods

The HTTP protocol defines many request methods that indicate the action you want to perform on a resource. The most common methods used in RESTful web APIs are:

⚠️ Important: The effect of a specific request method should depend on whether the resource is a collection or an individual item.

HTTP Methods Convention Table

Resource POST GET PUT DELETE
/customers Create a new customer Retrieve all customers Bulk update of customers Remove all customers
/customers/1 Error Retrieve details for customer 1 Update customer 1 if it exists Remove customer 1
/customers/1/orders Create a new order for customer 1 Retrieve all orders for customer 1 Bulk update of orders for customer 1 Remove all orders for customer 1
💡 Note: A web API doesn't need to implement all of the request methods. The methods it implements depend on the specific scenario.
GET

GET Requests

A GET request retrieves a representation of the resource at the specified URI. The body of the response message contains the details of the requested resource.

HTTP Status Codes

Status Code Reason
200 (OK) The method has successfully returned the resource
204 (No Content) The response body doesn't contain any content (e.g., search with no matches)
404 (Not Found) The requested resource can't be found
POST

POST Requests

A POST request should create a resource. The server assigns a URI for the new resource and returns that URI to the client.

⚠️ Important: For POST requests, a client shouldn't attempt to create its own URI. The client should submit the request to the URI of the collection, and the server should assign a URI to the new resource. If a client attempts to create its own URI and issues a POST request to a specific URI, the server returns HTTP status code 400 (BAD REQUEST).

In a RESTful model, POST requests are used to add a new resource to the collection that the URI identifies. However, a POST request can also be used to submit data for processing to an existing resource, without the creation of any new resource.

HTTP Status Codes

Status Code Reason
200 (OK) The method has done some processing but doesn't create a new resource
201 (Created) Resource created successfully. URI included in Location header
204 (No Content) The response body contains no content
400 (Bad Request) Invalid data in the request
405 (Method Not Allowed) POST not supported on this URI
PUT

PUT Requests

A PUT request should update an existing resource if it exists or, in some cases, create a new resource if it doesn't exist.

How PUT Works:

  1. The client specifies the URI for the resource
  2. The client includes a request body with a complete representation of the resource
  3. The client makes the request
  4. If a resource with this URI already exists, it's replaced. Otherwise, a new resource is created (if the route supports it)
⚠️ Idempotency Requirement: PUT requests must be idempotent, which means submitting the same request multiple times always results in the same resource being modified with the same values. In contrast, POST and PATCH requests aren't guaranteed to be idempotent.
💡 Tip: Consider implementing bulk HTTP PUT operations that can batch updates to multiple resources in a collection. This approach can help reduce chattiness and improve performance.

HTTP Status Codes

Status Code Reason
200 (OK) The resource was updated successfully
201 (Created) The resource was created successfully
204 (No Content) Resource updated successfully, but no content in response
409 (Conflict) Request couldn't be completed due to conflict with current state
PATCH

PATCH Requests

A PATCH request performs a partial update to an existing resource. The client specifies the URI for the resource. The request body specifies a set of changes to apply to the resource.

✅ Efficiency: PATCH can be more efficient than using PUT requests because the client only sends the changes and not the entire representation of the resource.

JSON Patch Formats

The two main JSON-based patch formats are:

1. JSON Merge Patch

The patch document has the same structure as the original JSON resource, but only includes the subset of fields that should be changed or added.

Original Resource
{ "name": "gizmo", "category": "widgets", "color": "blue", "price": 10 }
JSON Merge Patch
{ "price": 12, "color": null, "size": "small" }

Result: Updates price to 12, deletes color, adds size as "small". The values for name and category aren't modified.

⚠️ Limitation: Merge patch isn't suitable if the original resource can contain explicit null values, because of the special meaning of null in the patch document.

2. JSON Patch (RFC 6902)

More flexible - specifies changes as a sequence of operations to apply: add, remove, replace, copy, and test.

JSON Patch Example
[ { "op": "replace", "path": "/price", "value": 12 }, { "op": "remove", "path": "/color" }, { "op": "add", "path": "/size", "value": "small" } ]

HTTP Status Codes

Status Code Reason
200 (OK) The resource was updated successfully
400 (Bad Request) Malformed patch document
409 (Conflict) Patch document valid, but changes can't be applied to current state
415 (Unsupported Media Type) The patch document format isn't supported
DELETE

DELETE Requests

A DELETE request removes the resource at the specified URI.

HTTP Status Codes

Status Code Reason
204 (NO CONTENT) Resource deleted successfully. Response body contains no further information
404 (NOT FOUND) The resource doesn't exist

📦 Resource MIME Types

Resource representation is how a resource identified by URI is encoded and transported over HTTP in a specific format, such as XML or JSON. In the HTTP protocol, resource representation formats are specified by using media types (MIME types).

Content-Type Header

The Content-Type header in a request or response specifies the resource representation format.

POST Request with JSON
POST https://api.contoso.com/orders Content-Type: application/json; charset=utf-8 Content-Length: 57 {"Id":1, "Name":"Gizmo", "Category":"Widgets", "Price":1.99}
⚠️ If the server doesn't support the media type: Return HTTP status code 415 (Unsupported Media Type)

Accept Header

A client request can include an Accept header that contains a list of media types the client accepts from the server.

GET Request with Accept Header
GET https://api.contoso.com/orders/2 Accept: application/json, application/xml
⚠️ If the server can't match any listed media types: Return HTTP status code 406 (Not Acceptable)

📝 Summary

✅ HTTP Methods
Use GET, POST, PUT, PATCH, DELETE appropriately based on collections vs individual items.
✅ Asynchronous Operations
Return 202 Accepted with status endpoint for long-running operations.
✅ Pagination & Filtering
Implement limit/offset pagination and query-based filtering for efficiency.
✅ HATEOAS
Include hypermedia links in responses to enable client navigation.
✅ Versioning
Choose versioning strategy based on needs: URI, query string, header, or media type.
✅ Multitenancy
Design tenant isolation from the start using subdomain, header, or path-based approaches.
💡 Next Steps: Apply these patterns to build production-ready RESTful APIs. Focus on creating consistent, maintainable, and scalable API designs that follow industry best practices.

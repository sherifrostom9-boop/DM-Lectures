Module 03

Mastering HTTP Requests

Understanding execution flow, data retrieval, and error handling

1. Execution Flow: Sync vs Async

The Critical Decision: When making HTTP requests, the most important architectural choice is determining how the code waits for the server's response.

Synchronous Execution (Blocking)

Code runs sequentially. When a request is made, the program halts and waits for the server before moving to the next line.

  • Behavior: The UI may freeze or the browser may become unresponsive until data arrives.
  • Use Case: Rare in web dev; used in simple scripts where one step strictly requires the previous one.

Asynchronous Execution (Non-Blocking)

The code initiates the request and immediately continues executing the next lines without waiting.

  • Behavior: Patterns like "Promises" or "Async/Await" handle the response later. The UI stays responsive.
  • Use Case: The standard for modern apps (e.g., scrolling a feed while it loads).

☕ The Coffee Analogy:

  • Synchronous: Ordering coffee and standing at the counter staring at the barista until they hand it to you. You cannot do anything else.
  • Asynchronous: Ordering coffee, getting a buzzer (Promise), sitting down to read a book, and going back only when the buzzer rings.

2. Data Retrieval and Processing

Once a request is sent, the goal is to retrieve data and make it usable for the application.

The Request Methods (Verbs)

You must choose the correct method to tell the server what you want to do:

  • GET: Retrieve data (e.g., fetching a user profile).
  • POST: Send new data (e.g., submitting a signup form).
  • PUT/PATCH: Update existing data.
  • DELETE: Remove data.

The Payload and Headers

  • Headers: Meta-data sent with the request (e.g., Content-Type: application/json or Authentication tokens).
  • Body: The actual data being sent (used in POST/PUT).

Processing the Response

Raw data usually arrives as a string stream. You must parse it into a native data structure.

  • JSON (JavaScript Object Notation): The industry standard. It looks like text but is easily converted into objects/dictionaries.
  • XML: Older, more verbose, tag-based format.

3. Error Handling

Networks are unreliable. A robust application assumes requests will fail eventually and plans for it.

HTTP Status Codes

The server communicates the result via a 3-digit code:

  • 2xx (Success): Everything worked (e.g., 200 OK, 201 Created).
  • 4xx (Client Error): You made a mistake (e.g., 400 Bad Request, 404 Not Found).
  • 5xx (Server Error): The server crashed or is down (e.g., 500 Internal Server Error).

Handling Strategies

  • Check the Status: Always check if response.ok (in JS) before processing data.
  • Try/Catch Blocks: Wrap network calls in error handling blocks to catch failures like DNS errors or loss of connectivity.
  • User Feedback: Never show a raw error stack trace. Show a friendly message like "Something went wrong, please try again."

4. Summary Checklist

  • Async over Sync: Always keep apps responsive.
  • Parse Correctly: Match Content-Type (JSON).
  • Handle 404s/500s: Plan for server replies.

Event-Driven Programming

Understanding Synchronous vs Asynchronous Programming

1. Synchronous vs. Asynchronous

Synchronous (Blocking): Line-by-line execution. The UI freezes during heavy tasks.

Asynchronous (Non-Blocking): Tasks run in background. UI stays responsive.

2. The Event Loop

JavaScript is single-threaded but uses the Event Loop to handle async tasks.

Call Stack: Current execution.

Web APIs: Async processing.

Callback Queue: Waiting area for finished tasks.

3. Promises

A Promise represents a future value with 3 states: Pending, Fulfilled, Rejected.

4. Evolution of Async Patterns

Phase 1: Callbacks - Originally, we handled async tasks by passing a function (a callback) to be executed when the task finished.

The Issue: This nesting leads to "Callback Hell"—code that grows horizontally, becomes hard to read, and is difficult to debug.

Phase 2: Promises - A Promise is an object representing the eventual completion (or failure) of an asynchronous operation and its resulting value. It allows you to write asynchronous code that looks cleaner and flatter.

  • Pending: The initial state (operation not started or still in progress).
  • Fulfilled (Resolved): The operation completed successfully.
  • Rejected: The operation failed (e.g., network error).

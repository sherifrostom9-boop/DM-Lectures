The Core Pillars of Database Architecture

A database isn't just a collection of spreadsheets; it's a web of logic. To ensure your system doesn't collapse under data growth, we follow fundamental architectural principles that have stood the test of time.
Workshop Duration: 1.5 Hours (90 minutes)
Phase 1: The Blueprint (27 Minutes)
Phase 2: Building & Querying (63 Minutes)

Why Database Architecture Matters

  • Scalability: Handle growing data without performance degradation
  • Integrity: Ensure data accuracy and consistency across all operations
  • Performance: Optimize query execution times from seconds to milliseconds
  • Maintainability: Make future changes easy and safe to implement

Normalization: Eliminating Redundancy

Normalization is the process of structuring a database to reduce redundancy and improve data integrity. Each normal form builds upon the previous one, creating increasingly refined data structures.

1st Normal Form (1NF)

Rule: Atomic values only

No lists or multiple values in a single cell. Each cell must contain exactly one value.

Example:

❌ Phone: "555-1234, 555-5678"

✓ Separate rows or Phone1, Phone2

2nd Normal Form (2NF)

Rule: No partial dependencies

Everything must relate to the entire primary key, not just part of it.

Example:

In a composite key (StudentID, CourseID), course details shouldn't depend only on CourseID.

3rd Normal Form (3NF)

Rule: No transitive dependencies

Non-key attributes should not depend on other non-key attributes. All attributes must depend directly on the primary key.

Example:

❌ Storing both ZipCode and City (City depends on ZipCode, not UserID)

✓ Store only ZipCode, look up City in a separate table

The Golden Rule: Every piece of data should appear in exactly one place. If you can derive it, don't store it. If you need it frequently, cache it intelligently.

Constraints & Integrity

Database constraints are rules enforced by the database engine to maintain data quality and consistency. They act as guardrails, preventing invalid data from entering your system.

Primary Keys: The Unique DNA of a Record

A primary key uniquely identifies each row in a table. It must be:

  • Unique: No two rows can have the same primary key value
  • Not Null: Every row must have a primary key value
  • Immutable: Should never change once assigned
CREATE TABLE Employees (
  EmpID SERIAL PRIMARY KEY, -- Auto-incrementing unique identifier
  Name VARCHAR(100) NOT NULL,
  Email VARCHAR(100) UNIQUE -- Unique but can be changed
);

Foreign Keys: The Glue That Connects Tables

Foreign keys create relationships between tables, ensuring referential integrity. They prevent orphaned records and maintain data consistency.

CREATE TABLE Orders (
  OrderID SERIAL PRIMARY KEY,
  CustomerID INT REFERENCES Customers(CustomerID) ON DELETE CASCADE
);

-- ON DELETE CASCADE: Automatically delete orders when customer is deleted
-- ON DELETE SET NULL: Set CustomerID to NULL instead
-- ON DELETE RESTRICT: Prevent deletion if orders exist

Indexes: The Speed Boosters

Performance Impact:
• Without Index: O(n) - Linear search through all rows
• With Index: O(log n) - Binary search (logarithmic time)

Example: Finding one record in 1 million rows:
• No index: ~1,000,000 comparisons
• With index: ~20 comparisons

Project: School Management System

We're building a complete database schema that handles Employees, Projects, and their Assignments. This represents a classic many-to-many relationship pattern used in countless real-world applications.

Step 1: Create the Tables (DDL)

Copy and paste this into your SQL editor to build the foundation.

-- Create Employees Table
CREATE TABLE Employees (
  EmpID SERIAL PRIMARY KEY,
  Name VARCHAR(100) NOT NULL,
  Salary DECIMAL(10,2)
);

-- Create Projects Table
CREATE TABLE Projects (
  ProjectID SERIAL PRIMARY KEY,
  ProjectName VARCHAR(100)
);

-- Create Junction Table (Many-to-Many Resolver)
CREATE TABLE ProjectAssignments (
  AssignmentID SERIAL PRIMARY KEY,
  EmpID INT REFERENCES Employees(EmpID),
  ProjectID INT REFERENCES Projects(ProjectID),
  HoursWorked INT
);

Step 2: Seed the Data (DML)

Notice that 'David' hasn't been assigned a project yet—this is intentional for testing your LEFT JOINs.

-- Insert Employees
INSERT INTO Employees (Name, Salary) VALUES
('Alice', 75000),
('Bob', 68000),
('Charlie', 82000),
('David', 71000);

-- Insert Projects
INSERT INTO Projects (ProjectName) VALUES
('Alpha Portal'),
('Beta Dashboard'),
('Gamma API');

-- Assign Employees to Projects
INSERT INTO ProjectAssignments (EmpID, ProjectID, HoursWorked) VALUES
(1, 1, 120), -- Alice on Alpha Portal
(1, 2, 80), -- Alice also on Beta Dashboard
(2, 1, 95), -- Bob on Alpha Portal
(3, 3, 150); -- Charlie on Gamma API
-- Note: David (EmpID 4) has no assignments yet!

Step 3: Schema Metadata Summary

Table Primary Key Foreign Keys Purpose
Employees EmpID None Stores staff details and salary information
Projects ProjectID None Stores project metadata and names
ProjectAssignments AssignmentID EmpID, ProjectID Resolves Many-to-Many relationship

Practical Lab: Query Challenges

Now that we've built our schema, let's test it with real-world queries. These three challenges represent common business questions you'll encounter in production environments.

Challenge 1: The "Who's Busy?" Query

Business Question: Show me every employee, which projects they're working on, and how many hours they've logged.

SELECT
  e.Name AS Employee,
  p.ProjectName AS Project,
  pa.HoursWorked
FROM ProjectAssignments pa
JOIN Employees e ON pa.EmpID = e.EmpID
JOIN Projects p ON pa.ProjectID = p.ProjectID
ORDER BY e.Name;

Challenge 2: The "Ghost" Query

Business Question: Find employees who currently have zero project assignments.

SELECT e.Name AS Employee
FROM Employees e
LEFT JOIN ProjectAssignments pa ON e.EmpID = pa.EmpID
WHERE pa.AssignmentID IS NULL;

-- Expected Result: David
Key Concept: LEFT JOIN includes all rows from the left table (Employees) even if there's no match in the right table (ProjectAssignments). When there's no match, the right table's columns are NULL.

Challenge 3: The Budget Analysis

Business Questions:

  1. What's the total hours worked for each project?
  2. What's the average salary of employees on 'Alpha Portal'?
-- Total hours per project
SELECT
  p.ProjectName,
  SUM(pa.HoursWorked) AS TotalHours
FROM Projects p
LEFT JOIN ProjectAssignments pa ON p.ProjectID = pa.ProjectID
GROUP BY p.ProjectName
ORDER BY TotalHours DESC;

-- Average salary for Alpha Portal
SELECT AVG(e.Salary) AS AvgSalary
FROM Employees e
JOIN ProjectAssignments pa ON e.EmpID = pa.EmpID
JOIN Projects p ON pa.ProjectID = p.ProjectID
WHERE p.ProjectName = 'Alpha Portal';

The Three Levels of Modeling

Data modeling occurs at three distinct levels, each serving a specific purpose in the development process. Understanding these levels helps bridge the gap between business requirements and technical implementation.

Conceptual

Duration: 10 minutes

High-level business view. No technical jargon. Just "What things do we care about?"

Example: Customers, Orders, Products

Logical

Duration: 15 minutes

Defining the structure. We add attributes (Name, Date) and define relationships.

Example: Customer has Name, Email; Order has Date, Total

Physical

Duration: 15 minutes

The actual database implementation. We choose data types (INT, VARCHAR) and indexes.

Example: VARCHAR(50), DECIMAL(10,2)

Integration Phase: Total time for this lesson is approximately 1.5 hours

The Core Building Blocks

Every data model is constructed using three fundamental components. Think of these as the grammar of database design.
  • Entities: The "nouns" - objects or things we want to track (User, Product, Order)
  • Attributes: The "adjectives" - properties that describe entities (Email, Price, Date)
  • Relationships: The "verbs" - connections between entities (User buys Product, Order contains Product)

Common Relationship Types

1:1 (One-to-One)

Each record in Entity A relates to exactly one record in Entity B

Example: Person ↔ Passport

1:N (One-to-Many)

One record in Entity A relates to many records in Entity B

Example: Mother → Children

M:N (Many-to-Many)

Many records in Entity A relate to many records in Entity B

Example: Students ↔ Classes
Requires a junction table

Normalization: The "No Redundancy" Rule

Normalization is the process of organizing data to reduce redundancy and improve data integrity. The Golden Rule: Every non-key attribute must provide a fact about the key, the whole key, and nothing but the key.

The Normal Forms

  • 1st Normal Form (1NF): No multi-valued attributes - don't put three phone numbers in one cell. Each cell should contain only a single value.
  • 2nd Normal Form (2NF): No partial dependencies - everything in the table must depend on the entire primary key, not just part of it.
  • 3rd Normal Form (3NF): No transitive dependencies - don't store "City" if it depends on "Zip Code" instead of the "User_ID". Non-key attributes should depend only on the primary key.
Why Normalize?
  • Eliminates data redundancy
  • Ensures data consistency
  • Makes updates and maintenance easier
  • Reduces storage requirements

Physical Implementation (Code & Schema)

Creating the Tables

In this script, we use SERIAL for auto-incrementing IDs and REFERENCES to enforce data integrity.

CREATE TABLE Users (
  User_ID SERIAL PRIMARY KEY,
  Username VARCHAR(50) NOT NULL,
  Email VARCHAR(100) UNIQUE
);

CREATE TABLE Orders (
  Order_ID SERIAL PRIMARY KEY,
  User_ID INT REFERENCES Users(User_ID) ON DELETE CASCADE,
  Total_Amount DECIMAL(10,2)
);
Pro-Tips for Implementation:
  • Data Types: Use DECIMAL for money, never FLOAT (to avoid rounding errors)
  • Constraints: ON DELETE CASCADE automatically removes related records and keeps the database tidy
  • Indexing: Add an Index to frequently queried columns like email or product_id for speed
  • Naming: Use clear, consistent naming conventions (e.g., TableName_ID for primary keys)

Stress Test: Querying the Model

A well-designed model should make complex queries simple and performant. Let's test our model with real-world business questions.

Business Question

Query: "Find the total amount spent by 'JohnDoe'."

SELECT SUM(o.Total_Amount) AS Total_Spent
FROM Orders o
JOIN Users u ON o.User_ID = u.User_ID
WHERE u.Username = 'JohnDoe';

Understanding the Query

  • JOIN: Combines data from Users and Orders tables using the User_ID relationship
  • SUM: Aggregates all order amounts for the specified user
  • WHERE: Filters results to only include orders from 'JohnDoe'
Performance Tip: This query benefits from indexes on User_ID (foreign key) and Username. Without proper indexing, the database would need to scan every row to find matches.
